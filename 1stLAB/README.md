#### Аномалии
Аномалия вставки - логически не обозначены обязательные/опциональные поля. Так, сейчас обязательны все, что плохо, потому что клиент может не желать указывать свои контактные данные. Без них, то есть с NULL заполнить поля не получится.

Аномалия обновления - отсутствует обозначения уникальности клиента. ФИО клиентов могу совпадать, поэтому если возникнет потребность обновить контактную информацию, то нельзя в команде UPDATE привязаться к имени, иначе обновим информацию для разных клиентов, что критически нельзя делать.

Аномалия удаления - отсутствует структурированная информация о каталоге товаров. Так, мы не можем сейчас использовать данную таблицу для хранения ассортимента, так как, если мы решим какой-то товар не продавать больше, то удалять его из таблицы, во-первых, неполучится, так как он соседствует с другим товаром в одной строке, а, во-вторых, если товар будет указан один, то при удалении его удалим и клиента, удалим историю о заказе, что нельзя.

### Часть 2. Нормализация до 3NF
#### 1NF
Файл - [1NF](https://github.com/danilaercegovac/Data-Base-2026/blob/main/1stLAB/1NF.sql). При создании поменял тип данных у цены и количества продуктов с TEXT на NUMERIC и INTEGER соответственно, так как в рамках нормализации к первой форме мы разделим продукты на разные строки, то есть к каждому продукту будет принадлежать своё значение цены и количества. В INSERT скрипте нашёл функцию string_to_array - она преобразует строку в массив по разделителю, то есть, разделив в нашем случае строку поля продукта, поле продукта приобрела свойство массивность, то есть поле продукта, имевшее порядок 1, сохранила порядок один, но приобрела подпорядки 1.1 и 1.2, если в строке было два значения через запятую. А функция unnest() развернула массив в набор строк, то есть присвоила значениям подпорядков собственный порядок и продублировала не массивную информацию для каждой строки.

#### 2NF
Файл - [2NF](https://github.com/danilaercegovac/Data-Base-2026/blob/main/1stLAB/2NF.sql). Создал таблицы customers, orders, order_items, products, указали в каждом первичные и внешние ключи. Добавил в них соответствующие данные.

Традиционной частичной зависимости здесь нет, так как существует только один первичный ключ order_id (он не составной). Тем не менее проблемы были решены следующие: контактная информация принадлежала order id, хотя логически принадлежит клиенту (здесь за клиентский ключ можно считать ФИО или почту). Также продуктовые поля (количество, название, цена) принадлежат order id, но логически относятся к составляющей заказа, строке. Разделив сущности, мы устранили такие зависимости.

### Часть 6. Сравнительная таблица OLTP vs OLAP

| Характеристика | OLTP (PostgreSQL) | OLAP (ClickHouse) |
|----------------|-------------------|--------------------|
| Модель хранения | строковая | колоночная |
| Типичный запрос | Короткие, простые транзакции | Сложные аналитические запросы с агрегациями |
| Нормализация | Высокая: 3NF и выше | Денормализация: плоские таблицы или минимальное число связей |
| Транзакции | Полная поддержка ACID | Ограниченная поддержка транзакций |
| Вставка | Построчно | Пакетная |
| Обновление/удаление | Частые точечные операции | Минимальные или отсутствующие |
| Масштабирование | Вертикальное | Горизонтальное |
| Типичный use case | Операционные системы: веб‑приложения, CRM, ERP, банковские транзакции, корзины покупок | Аналитические системы: BI‑отчёты, дашборды, машинное обучение, анализ логов, поведенческая аналитика |
