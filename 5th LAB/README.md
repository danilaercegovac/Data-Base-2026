## Описание работы

### Часть 1. Выполнение работы с ManticoreSearch

1. На домашнем сервере были подняты контейнеры с ManticoreSearch и PostgreSQL.
2. После запуска сервисов была проверена доступность подключений. Результаты проверки сохранены в файле `connectivity.txt`.
3. В ManticoreSearch был создан RT-индекс для хранения и поиска тестовых данных.
4. С помощью Python-скрипта был сгенерирован набор из 100000 записей.
5. Далее были выполнены 5 поисковых запросов из файла `02_search_queries.sql`.  
   В итоговых txt-файлах приведены сами команды, результаты их выполнения и время обработки. Для HTTP-запросов использовался `curl`.
6. Затем в PostgreSQL была создана таблица `pg_products`, которая была заполнена данными через скрипт `data_migration.py`.
7. Для PostgreSQL дополнительно была добавлена колонка `tsv` и создан GIN-индекс. После этого были выполнены 5 запросов, аналогичных запросам к ManticoreSearch.

### Часть 2. Объяснение использованных решений

#### Настройка RT-индекса и отличие от plain-индекса

В индексе ManticoreSearch использовалась настройка `morphology='stem_enru'`. Она включает стемминг для английского и русского языков, то есть позволяет приводить слова к их основе. Благодаря этому поиск может находить не только точное совпадение, но и разные формы одного и того же слова.

Параметр `min_word_len='2'` задаёт минимальную длину слова, которое будет попадать в индекс. Это помогает не индексировать слишком короткие и часто бесполезные элементы. Настройка `html_strip='1'` нужна для удаления HTML-тегов из текста перед индексацией, что особенно полезно при работе с веб-контентом.

RT-индекс отличается от plain-индекса тем, что он рассчитан на обновление данных в реальном времени. Plain-индексу для обновления содержимого требуется переиндексация, поэтому он менее удобен для данных, которые часто меняются. В данной работе RT-индекс подходит лучше, так как данные можно загружать и изменять без полной перестройки индекса.

#### Загрузка данных в ManticoreSearch

Для заполнения индекса был использован скрипт `scripts/load_products.py`. После его выполнения был сделан проверочный запрос:

```sql
SELECT COUNT(*) FROM products;
```

Он вернул следующий результат:

```text
+----------+
| count(*) |
+----------+
|   100000 |
+----------+
```

Это подтверждает, что в индекс было успешно загружено 100000 записей.

#### Фасетный поиск

Фасетный поиск особенно полезен для интернет-магазинов и каталогов товаров. Он позволяет не только выполнить поиск, но и сразу получить распределение найденных товаров по категориям, брендам, цветам и другим параметрам.

Основные преимущества фасетов:

- пользователь сразу видит количество товаров в каждой группе;
- не нужно выполнять несколько отдельных запросов для подсчётов;
- поиск выполняется один раз, а группировка применяется уже к найденному набору данных;
- фильтрация становится удобнее и быстрее для конечного пользователя.

### Сравнение ManticoreSearch и PostgreSQL

#### Время выполнения запросов

| Тип запроса                                   | ManticoreSearch            | PostgreSQL                                                           |
| -------------------------------------------- | -------------------------- | -------------------------------------------------------------------- |
| Базовый поиск `wireless bluetooth headphones` | 5 мс                       | 14.8 мс                                                              |
| Фразовый поиск `"noise cancelling"`           | 6 мс                       | 7.4 мс                                                               |
| Proximity поиск `"portable speaker"~3 / <->`  | 6 мс, расстояние до 3 слов | 7.4 мс, но аналог неполный, так как учитываются только соседние слова |
| Поиск с фильтрацией `laptop + цена + рейтинг` | 5 мс                       | 16.9 мс                                                              |
| Поиск по JSON `tags.color = 'black'`          | 6 мс                       | 7.7 мс                                                               |

По результатам замеров видно, что ManticoreSearch в большинстве случаев обрабатывал поисковые запросы быстрее. Особенно заметная разница проявилась в базовом поиске и поиске с дополнительной фильтрацией.

#### Функциональное сравнение

| Характеристика                      | ManticoreSearch                                    | PostgreSQL `tsvector`                                                |
| :---------------------------------- | :------------------------------------------------- | :------------------------------------------------------------------- |
| **Релевантность `BM25` vs `ts_rank`** | Использует вероятностную модель ранжирования       | Ранжирование основано на частоте терминов                            |
| **Морфология из коробки**           | Есть `stem_enru` для английского и русского языков | В данном варианте использовалась только английская конфигурация       |
| **Фасетный поиск**                  | Поддерживается напрямую                            | Прямой поддержки нет, можно заменить группировкой                    |
| **JSON-атрибуты**                   | Поддерживается поиск по JSON                       | Есть тип `JSONB` и операции по нему                                  |
| **Транзакции**                      | Не поддерживаются в классическом виде              | Полноценная поддержка транзакций                                     |
| **Когда использовать**              | Поиск, каталоги, e-commerce-фильтры, лог-аналитика | Основное хранение данных, пользователи, заказы, финансовые операции, OLTP |

ManticoreSearch удобнее использовать как специализированный поисковый движок. Он хорошо подходит для случаев, где важны скорость поиска, релевантность, фасеты и фильтрация. PostgreSQL, в свою очередь, лучше подходит для хранения основных данных приложения, особенно если нужны транзакции, согласованность и надёжность записи.

#### Различия при UPDATE

При обновлении данных между ManticoreSearch и PostgreSQL есть несколько важных отличий.

1. В ManticoreSearch операции обновления не являются транзакционными, поэтому откатить изменение после выполнения `UPDATE` нельзя.
2. При изменении данных в ManticoreSearch индексы могут требовать дополнительной актуализации или перестроения. В PostgreSQL индексы обновляются синхронно вместе с данными в рамках одной транзакции.
3. PostgreSQL использует журнал предзаписи, поэтому изменения сначала фиксируются в WAL, а затем записываются на диск. Это повышает надёжность хранения данных. В ManticoreSearch таких гарантий в случае внезапного сбоя меньше.

В итоге ManticoreSearch лучше рассматривать как инструмент для быстрого поиска и фильтрации, а PostgreSQL — как основную транзакционную базу данных. В реальных проектах эти системы часто могут использоваться вместе: PostgreSQL хранит основную информацию, а ManticoreSearch отвечает за быстрый полнотекстовый поиск.
