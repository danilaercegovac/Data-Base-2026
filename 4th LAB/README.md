# 4 лабораторная работа
## Описание работы

### Часть 1. Кластер 2x2

- Работа выполняется через Docker Compose.
- По заданию поднимаем кластер ClickHouse из 2 шардов, в каждом шарде по 2 реплики.
- Для координации реплик используются 3 узла ClickHouse Keeper.

Структура узлов ClickHouse:

```text
shard1: ch-s1-r1, ch-s1-r2
shard2: ch-s2-r1, ch-s2-r2
```

Keeper-узлы:

```text
keeper1
keeper2
keeper3
```

Для удобного подключения к контейнерам можно добавить алиасы:

```bash
alias ch-s1r1='docker exec -it ch-s1-r1 clickhouse-client'
alias ch-s1r2='docker exec -it ch-s1-r2 clickhouse-client'
alias ch-s2r1='docker exec -it ch-s2-r1 clickhouse-client'
alias ch-s2r2='docker exec -it ch-s2-r2 clickhouse-client'
```

Запускаем контейнеры:

```bash
docker compose up -d
```

Проверяем, что контейнеры поднялись:

```bash
docker ps
```

Проверяем описание кластера в ClickHouse:

```bash
docker exec -i ch-s1-r1 clickhouse-client --query "SELECT * FROM system.clusters WHERE cluster = 'cluster_2x2' FORMAT Vertical" > checks/cluster_info.txt
```

Ожидаем, что в результате будут отображены 2 шарда и 4 реплики.

### Часть 2. Локальные и распределённые таблицы

Предметная область лабораторной — события пользовательской аналитики, то есть clickstream.

Сначала создаём локальную таблицу на всех узлах кластера:

```bash
docker exec -i ch-s1-r1 clickhouse-client < sql/01_create_local.sql
```

Файл `sql/01_create_local.sql` содержит создание таблицы `events_local` на кластере `cluster_2x2`.

После этого создаём распределённую таблицу:

```bash
docker exec -i ch-s1-r1 clickhouse-client < sql/02_create_distributed.sql
```

Распределённая таблица `events_distributed` использует локальную таблицу `events_local` и распределяет данные по ключу:

```sql
xxHash64(user_id)
```

Ключ выбран по `user_id`, потому что в аналитике событий часто нужны запросы по конкретному пользователю: история действий, количество событий, сессии, переходы по страницам. Если все события одного пользователя попадают на один шард, то ClickHouse не нужно дополнительно собирать эти строки с разных шардов.

`event_date` хуже подходит для ключа шардирования, потому что активность пользователей по датам может быть неравномерной. Например, в один день событий может быть намного больше, чем в другой. Тогда часть шардов будет перегружена.

`rand()` тоже не подходит, хотя он может распределять строки равномерно. Проблема в том, что события одного и того же пользователя будут случайно разлетаться по разным шардам. Из-за этого запросы по `user_id` станут менее эффективными.

### Часть 3. Наполнение и проверка распределения

Для наполнения таблицы был подготовлен скрипт:

```text
scripts/generate_clickstream.sh
```

Запуск:

```bash
bash scripts/generate_clickstream.sh
```

Скрипт вставляет 2 000 000 строк в `events_distributed`. Данные генерируются через `numbers`, `rand`, `generateUUIDv4` и набор тестовых типов событий.

Проверяем общее количество строк:

```bash
docker exec -i ch-s1-r1 clickhouse-client --query "SELECT count() FROM events_distributed"
```

Проверяем распределение данных по репликам и шардам:

```bash
bash scripts/collect_checks.sh
```

Основной результат сохраняется в файл:

```text
checks/data_distribution.txt
```

В проверке фиксируются:

- количество строк на локальных таблицах;
- количество уникальных пользователей на каждом узле;
- общее количество строк через `events_distributed`;
- проверка того, что один `user_id` не распался на несколько шардов.

Для проверки пользователя используется запрос, который группирует данные по `user_id` и считает количество разных хостов для каждого пользователя. Если максимум равен `1`, значит события одного пользователя находятся на одном шарде.

### Часть 4. Запросы через Distributed

Запросы для этой части находятся в файле:

```text
sql/04_queries.sql
```

Запуск:

```bash
docker exec -i ch-s1-r1 clickhouse-client < sql/03_user_dict.sql
docker exec -i ch-s1-r1 clickhouse-client < sql/04_queries.sql > checks/distributed_queries.txt
```

#### Глобальный COUNT

Первый запрос считает все строки через распределённую таблицу:

```sql
SELECT count() FROM events_distributed;
```

Далее результат сравнивается с суммой строк из локальных таблиц через `clusterAllReplicas`.

#### GROUP BY по шардированному ключу

Запрос top-10 пользователей:

```sql
SELECT
    user_id,
    count() AS events
FROM events_distributed
GROUP BY user_id
ORDER BY events DESC
LIMIT 10;
```

Такой запрос выполняется эффективно, потому что `user_id` является ключом шардирования. Все события одного пользователя лежат на одном шарде, поэтому ClickHouse не нужно собирать одного и того же пользователя из разных мест.

#### GROUP BY без шардированного ключа

Запрос top-10 страниц:

```sql
SELECT
    page_url,
    count() AS visits
FROM events_distributed
GROUP BY page_url
ORDER BY visits DESC
LIMIT 10;
```

`page_url` не участвует в ключе шардирования. Одна и та же страница может встретиться на любом шарде, поэтому ClickHouse должен собрать промежуточные результаты с разных узлов и объединить их. Из-за этого появляется дополнительный обмен данными между шардами.

#### JOIN со справочником

Для JOIN создаётся справочная таблица `user_dict`:

```text
sql/03_user_dict.sql
```

В ней есть поля:

```text
user_id, name, segment
```

Пример запроса:

```sql
SELECT
    d.segment,
    count() AS events
FROM events_distributed AS e
INNER JOIN user_dict_distributed AS d ON e.user_id = d.user_id
GROUP BY d.segment
ORDER BY events DESC;
```

Обычный JOIN на распределённых таблицах может приводить к broadcast JOIN. Это значит, что правая часть JOIN может рассылаться или выполняться на разных шардах, из-за чего появляется лишняя сетевая нагрузка.

Один из способов уменьшить проблему — использовать `GLOBAL JOIN` или заранее сделать справочник доступным одинаково на всех нужных узлах. В таком случае правая часть запроса сначала подготавливается, а потом отправляется на шарды в более предсказуемом виде.

### Часть 5. Добавление третьего шарда

После проверки кластера 2x2 добавляется третий шард с двумя репликами:

```text
shard3: ch-s3-r1, ch-s3-r2
```

Для этого нужно:

- добавить два новых сервиса ClickHouse в `docker-compose.yml`;
- добавить третий шард в `remote_servers`;
- создать макросы для новых реплик;
- перезапустить контейнеры;
- пересоздать распределённую таблицу, чтобы она знала о новом составе кластера.

Пример обновлённого конфига для кластера 3x2 лежит в файле:

```text
config/after_add_third_shard/cluster_3x2.xml
```

После обновления конфигурации можно вставить новые данные:

```bash
docker exec -i ch-s1-r1 clickhouse-client < sql/05_insert_after_third_shard.sql
```

Результат проверки сохраняется в файл:

```text
checks/reshard_demo.txt
```

Важно: старые данные автоматически не переезжают на третий шард. ClickHouse начинает использовать новый шард только для новых вставок через распределённую таблицу.

Из-за этого может возникнуть ситуация, когда старые события пользователя лежат на одном шарде, а новые после изменения конфигурации начинают попадать на другой. Это ухудшает локальность данных и может замедлить запросы по пользователю.

Для ребалансировки можно выполнить ручной перенос данных:

- создать временную таблицу или использовать вставку `INSERT SELECT`;
- посчитать новый ключ шардирования уже с учётом трёх шардов;
- перенести строки на нужные шарды;
- проверить количество строк и пользователей;
- после проверки удалить старые копии.

В рамках этой работы реализация ребалансировки не выполнялась, так как по заданию достаточно описать подход.

## Вывод

В лабораторной работе был подготовлен шардированный кластер ClickHouse 2x2 с репликацией через ClickHouse Keeper. Были созданы локальные и распределённые таблицы для clickstream-данных, выполнена генерация 2 000 000 строк и проверено распределение данных по шардам.

Ключ `xxHash64(user_id)` подходит для данной предметной области, потому что события одного пользователя остаются на одном шарде. Это делает запросы по пользователям более эффективными. При группировке по полю, которое не является ключом шардирования, например `page_url`, ClickHouse вынужден собирать и объединять данные с разных шардов.

Также была рассмотрена проблема broadcast JOIN и описан вариант её решения через `GLOBAL JOIN` или корректно распределённый справочник. При добавлении третьего шарда новые данные начинают распределяться уже по новой конфигурации, но старые данные автоматически не переносятся, поэтому для полного выравнивания требуется отдельная ребалансировка.
